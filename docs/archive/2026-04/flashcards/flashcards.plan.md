# Plan: flashcards 암기카드 콘텐츠 보강

## 개요

정보처리기사 실기 시험 대비 암기카드(flashcards)의 학습 효과를 높이기 위해, 현재 너무 짧고 이해하기 어려운 `definition`(설명)을 쉽고 상세하게 보강하고, 시험 출제 경향과 암기 요령을 알려주는 `tip`(암기 TIP) 및 `mnemonic` 필드를 전체 카드에 추가한다.

## 현황 분석

### 대상 파일 (4개, 총 523장)

| 파일 | 카드 수 | 짧은 설명(50자 미만) | tip 있음 | tip 없음 |
|------|---------|---------------------|----------|----------|
| flashcards-se.json | 241 | 176 (73%) | 3 | 238 |
| flashcards-db.json | 93 | 67 (72%) | 0 | 93 |
| flashcards-os.json | 86 | 65 (76%) | 0 | 86 |
| flashcards-network.json | 103 | 78 (76%) | 0 | 103 |
| **합계** | **523** | **386 (74%)** | **3** | **520** |

### 현재 문제점

1. **설명 부족**: 전체 카드의 74%가 50자 미만의 극히 짧은 설명으로, 개념을 이해하기 어려움
   - 예: `FCFS` → "먼저 도착한 순서" (8자)
   - 예: `DDL` → "CREATE, ALTER, DROP, TRUNCATE\n데이터 정의" (키워드만 나열)
   - 예: `ARP` → "논리주소→물리주소" (7자)

2. **암기 TIP 부재**: 520/523장(99.4%)에 tip이 없음
   - 시험에 어떤 식으로 출제되는지 알 수 없음
   - 자주 나오는 오답 패턴이나 함정을 미리 알 수 없음

3. **기존 좋은 예시** (flashcards-se.json에 3장만 존재):
   - 스크럼 카드: "스크럼 마스터가 팀원을 통제한다"가 오답인 이유 설명
   - XP 가치 카드: "전문성(Expertise)이 5대 가치"라는 오답 패턴 경고
   - 이 수준의 tip을 전체 카드에 확대 적용 필요

## 목표

### 1. definition(설명) 보강
- **짧은 설명**: 개념의 "왜/어떻게"를 추가하여 이해 가능한 수준으로 확장
- **이미 긴 설명**: 기존 구조 유지, 오탈자 수정 및 이해도 개선만 진행
- **보강 기준**:
  - 용어의 한글 풀이 + 핵심 개념 1~2줄
  - 비유나 실생활 예시 활용 (가능한 경우)
  - 암기 키워드(두문자어) 포함 (가능한 경우)
  - 관련 용어와의 차이점 명시 (relatedTerms가 있는 경우)

### 2. tip(암기 TIP) 추가
- **전체 520장**에 `tip` 필드 추가
- **tip 내용 기준**:
  - 시험에서 자주 출제되는 형태 안내
  - 헷갈리기 쉬운 오답 패턴 경고
  - 비슷한 개념과의 구분법
  - 암기 요령 (두문자어, 연상법 등)
- **mnemonic 필드**: `"암기 TIP"` 값으로 통일 (기존 3장과 동일 형식)

### 3. 데이터 무결성 유지
- 기존 `id`, `categoryId`, `subcategory`, `source`, `sourceQuestionId`, `tags`, `relatedTerms` 등의 필드는 절대 변경하지 않음
- JSON 형식 유효성 보장

## 작업 범위 및 우선순위

### Phase 1: flashcards-se.json (소프트웨어 공학) - 241장
- 가장 카드 수가 많고, 이미 3장에 tip 예시가 있어 기준 수립에 적합
- 방법론, 설계, 테스트, UML 등 다양한 서브카테고리

### Phase 2: flashcards-db.json (데이터베이스) - 93장
- SQL 분류(DDL/DML/DCL/TCL), 정규화, 키, 트랜잭션 등
- 짧은 설명이 많아 보강 효과가 큼

### Phase 3: flashcards-os.json (운영체제) - 86장
- 스케줄링, 메모리, 프로세스, 교착상태 등
- 계산 문제 관련 카드는 공식 + 예시 추가

### Phase 4: flashcards-network.json (네트워크) - 103장
- OSI 7계층, 프로토콜, IP, 보안 등
- 프로토콜 간 차이점 비교가 핵심

## 보강 형식 예시

### Before (현재)
```json
{
  "id": "card-memorize-os_001",
  "term": "FCFS",
  "definition": "먼저 도착한 순서"
}
```

### After (보강 후)
```json
{
  "id": "card-memorize-os_001",
  "term": "FCFS",
  "definition": "FCFS (First Come First Served, 선입선출)\n\n준비 큐에 먼저 도착한 프로세스부터 순서대로 CPU를 할당하는 스케줄링 방식\n\n[특징]\n• 비선점형: 한번 CPU를 잡으면 끝날 때까지 뺏기지 않음\n• 구현이 가장 단순 (큐 하나면 됨)\n• 단점: 호위 효과(Convoy Effect) — 긴 작업이 앞에 있으면 뒤의 짧은 작업들이 오래 기다림",
  "mnemonic": "암기 TIP",
  "tip": "FCFS는 '비선점형'이라는 점과 '호위 효과(Convoy Effect)'를 묻는 문제가 자주 나옵니다. SJF와 혼동하는 보기(실행시간 짧은 순)가 오답으로 등장하니, FCFS=도착 순서임을 확실히 구분하세요!"
}
```

## 제약 조건

- 외부 API나 서비스 사용 불가 (로컬 JSON 직접 편집)
- 기존 앱 코드(컴포넌트, 스토어 등)는 수정 불필요 — 데이터만 보강
- 정보처리기사 실기 시험 범위에 맞는 정확한 내용만 작성
- JSON 파싱 에러가 발생하지 않도록 특수문자(따옴표 등) 이스케이프 처리

## 성공 기준

1. 전체 523장의 definition이 개념 이해에 충분한 수준으로 보강됨
2. 전체 523장에 `tip`과 `mnemonic` 필드가 존재함
3. 모든 JSON 파일이 유효한 형식을 유지함
4. 기존 필드(id, tags, relatedTerms 등)가 변경되지 않음
