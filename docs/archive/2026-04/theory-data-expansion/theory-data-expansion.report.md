### 2.2 Design (설계) 단계

**문서**: 별도 디자인 문서 없음 (데이터 기반 기능으로 구조 이미 확정)

#### 설계 의사결정

1. **데이터 구조 유지**
   - 기존 JSON 파일 포맷 유지
   - 필드: id, categoryId, subcategory, type, difficulty, question, answer, explanation, choices, source, sourceFile, tags
   - 변경 없음

2. **Subcategory 확장 전략**
   ```
   theory-se subcategories:
   - 개발방법론 (기존)
   - 요구사항, UML, 디자인패턴, 비용산정, 아키텍처, 프로젝트관리 (기존 + 확장)
   - 통합구현, 서버프로그램, 인터페이스, 화면설계, 테스트, 패키징 (신규)
   ```

3. **문제 분류 기준**
   - 이론: type="theory", 4지선다 선택지 포함
   - 암기: type="memorize", 선택지 없음 (카드형)
   - 난이도: 1(기본) / 2(응용) / 3(복합)

4. **기출 문제 표시**
   - tags 필드에 (기출) 포함
   - 우선순위 최고 (Plan 단계에서 확정)

- 기출 문제 우선순위 준수 (30개 모두 포함)

✅ **체계적인 문제 검증**
- 중복 문제 0개
- 선택지 누락 0개
- 난이도 분포 적절

### 5.2 개선 필요 사항

⚠️ **Subcategories 앱 UI 반영 미완**
- **상황**: categories.json에는 등록되었으나 앱 UI 필터에 모두 표시되지 않음
- **이유**: 앱 UI 컴포넌트가 모든 subcategory를 렌더링하지 못할 수 있음
- **해결 방안**:
  1. 우선순위 높은 subcategory 5-8개만 선택해서 categories.json에 유지
  2. 나머지는 향후 앱 UI 개선 후 추가
  3. 또는 앱 필터 UI를 동적으로 개선하여 모든 subcategory 표시
- **영향도**: 낮음 (데이터는 정상 저장되어 있으므로 앱 개선 후 자동 지원 가능)

### 5.3 향후 적용 사항

📌 **다음 대규모 데이터 확장 시 참고**

1. **Early Planning + Iteration**

   - 구현 중 더 나은 방안을 발견하면 즉시 Plan을 v2로 업데이트
